# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A multi-tenant ("SaaS") backend for a university club management system (think: student clubs, advisors, join requests, announcements) — one deployment serves many universities, tenant-scoped by `universityId`. Built with Bun + Hono + Drizzle ORM + PostgreSQL, with Redis/BullMQ for background jobs.

Code comments, commit-style notes, and API-facing error messages throughout the codebase are written in Turkish. Preserve this convention when editing existing files.

## Commands

Runtime is **Bun**, not Node — use `bun`/`bunx`, not `npm`/`npx`.

```sh
bun install                # install deps
bun run dev                # start dev server with --hot reload (src/index.ts), http://localhost:3000
docker-compose up -d       # start local Postgres (5432) + Redis (6379) — required before `dev`

bun run db:generate        # drizzle-kit generate — create SQL migration from schema.ts changes
bun run db:migrate         # drizzle-kit migrate — apply pending migrations
bun run db:push            # drizzle-kit push  — push schema directly without a migration file
bun run db:seed            # bun run src/db/seed.ts — seeds 3 universities (tenant-isolation scenarios), the 9-role RBAC catalog + bundles, sample users for every role, clubs in every status, applications, memberships
bun run db:reset           # drop + generate + migrate + seed, in one shot (interactive — drizzle-kit drop prompts)

bun run typecheck          # tsc --noEmit — the check CI runs; must pass before pushing

bun run test:setup         # provision the isolated `uniclub_test` DB (drop + migrate + seed) — run once before `test`
bun run test               # bun test — integration tests (Hono app.request against real Postgres/Redis)
bun run test:all           # test:setup + test, in one shot
```

Quality gates are `typecheck` **and** `test` (both run in CI — see `.github/workflows/ci.yml`). There is still **no lint script**, so don't assume `bun run lint` exists. The tests are integration-style: they exercise the full middleware chain (auth, RBAC `guard()`, tenant scope) via Hono's `app.request()` against a **separate `uniclub_test` database** and Redis DB index 1, so they never touch dev/CI data. `tests/provision.ts` drops+recreates+seeds that DB each run (deterministic seed → tests key off fixed emails); `tests/setup.ts` is the `bunfig.toml` preload that repoints env **before** `src/config/env.ts` reads it. `app` is exported from `src/index.ts` for this — importing it does not start `Bun.serve` (that only happens when the file is the entrypoint).

**Branching**: `main` is protected (PR + green CI + 1 approval); daily work branches off `develop`. Never commit straight to `main`. See `CONTRIBUTING.md`.

**Environments are not branches.** `main` and `develop` hold identical files. Dev/prod differ by env vars and by build scope — `.dockerignore` keeps `docs/`, `.github/` and dev tooling out of the production image (`Dockerfile`, `docker-compose.prod.yml`).

Required env vars (validated at startup, see below): `PORT`, `NODE_ENV`, `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`. Mail-related vars (`SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASS`, `MAIL_FROM`, `APP_URL`) and rate-limit vars (`RATE_LIMIT_DISABLED`, `TRUST_PROXY`) all have dev defaults. Boolean env vars go through the `envBoolean` helper in `env.ts`, **not** `z.coerce.boolean()` — the latter turns the string `"false"` into `true`.

`docker-compose` brings up Postgres (5432), Redis (6379) and **Mailpit** — a fake SMTP server on 1025 with a web inbox at http://localhost:8025 where the verification emails land.

## Architecture

**Layering, per feature.** Code lives under `src/features/<feature>/`, one file per concern:
- `*.routes.ts` — Hono route definitions, request validation via `zValidator` + a zod schema, calls into the service, shapes the `{ success, message, data }` JSON response.
- `*.service.ts` — business rules (the numbered-comment-per-rule style used in `auth.service.ts` is the established pattern). Throws plain `Error`s with user-facing Turkish messages; routes catch them and turn them into HTTP responses.
- `*.repository.ts` — the only layer that talks to `db` / `schema` directly. Services never import `db` directly.
- `*.schema.ts` — zod request schemas + inferred DTO types.
- `*.types.ts` — Drizzle-inferred entity types (`InferSelectModel`) and payload interfaces used by the repository.

**Implemented features** (all under `src/features/`): `auth` (register/login/verify/me + full RBAC: role & permission catalog, role assignment, per-user permission overrides), `users` (self-service profile/password/my-clubs/my-applications/my-advised-clubs + `GET /me/permissions` effective set), `university` (public university/faculty/department/domain reads + granular guarded writes), `admin` (tenant-scoped user/club/application/advisor management + club content moderation), `clubs` (browse/join/leave, creation applications, club-internal membership management via `club.middleware`, profile/contact-links), `announcements` & `gallery` (per-club sub-resources), `notifications` (persisted notifications + real-time WebSocket delivery, see below), `audit` (append-only audit trail, auto-captured by `guard()`, see below). The frontend-facing endpoint reference for the management surface is `docs/FRONTEND_YONETIM.md`; the RBAC model design + scenarios live in `docs/yonetim/`.

**Database** (`src/db/`):
- `schema.ts` is the single source of truth for all tables, built with Drizzle's `pgTable`/`pgEnum`. Tenant hierarchy: `universities` → `universityDomains` (a university can own multiple email domains, each tagged `student`/`staff`) → `faculties` → `departments`. `users` denormalizes `universityId` directly (intentional, for fast tenant-scoped queries) but does *not* denormalize it onto `departments` (deliberately — reached via `faculty → university`).
- Two permission layers: a global claim-based layer (`roles`, `permissions`, `rolePermissions`, `userRoles`, `userPermissions`) and a per-club layer (`clubMembers.role`: member/officer/president). Don't conflate the two. The global layer ships an **enterprise 9-role model** seeded by `seed.ts` (`super_admin`, `platform_support`, `university_admin` — the former `admin`, renamed — `student_affairs`, `academic_affairs`, `content_moderator`, `auditor`, `advisor`, `student`); `roles.universityId` is nullable so a role is either a global template (`null`) or tenant-scoped. `userPermissions.granted` is a per-user grant(`true`)/deny(`false`) override applied on top of the union of role permissions (deny wins). See `docs/yonetim/06-rol-mimarisi-yeniden-tasarim.md` for the role→permission bundles and the escalation-safety rules on tenant-scoped role management.
- `clubs` has three distinct many-to-many relationships to `users` (creator via `createdBy`, advisors via `clubAdvisors`, members via `clubMembers`) — `relations.ts` gives each an explicit `alias` (`creator_club`/`advisor_club`/`member_club`) because Drizzle's relational query API requires disambiguation when two tables are linked more than once.
- `clubApplications` + `clubApplicationApprovals` model an extensible multi-step approval chain (`step` int) — currently only step 1 (advisor) is used; adding a second approval stage (e.g. student affairs office) means inserting a `step: 2` row, not a schema change.
- `relations.ts` uses Drizzle's newer `defineRelations` (v2 relational API, `r.one`/`r.many`), not the older `relations()` helper — repositories query via `db.query.<table>.findFirst({ where: {...} })` object-style `where`, not the legacy callback/operator style.
- Migrations: `drizzle.config.ts` points `out` at `src/db/migrations` — that is the only migrations directory. (A stale top-level `drizzle/` directory from before that config was set has been removed and is `.gitignore`d, so it can't come back.)

**Config** (`src/config/env.ts`): all env access must go through the zod-validated `env` export — never read `process.env` directly in app code. Invalid/missing env vars throw immediately at import time with a Turkish message identifying the field. `drizzle.config.ts` is the one exception (reads `process.env.DATABASE_URL` directly, since drizzle-kit runs outside the app's module graph).

**Auth**: JWT via `hono/jwt` (HS256, 7-day expiry), payload is `{ userId, universityId, exp }` (`src/shared/utils/jwt.util.ts`). Passwords hashed with `Bun.password` (bcrypt) in `src/shared/utils/password.util.ts`. `authMiddleware` (`src/core/auth/auth.middleware.ts`) reads `Authorization: Bearer <token>`, verifies it, and sets a typed `user` context variable (`Variables` type) that downstream handlers read via `c.get("user")`. Registration infers the tenant from the email domain (`user@<domain>` looked up in `universityDomains`) rather than an explicit tenant selector — there is no domain match, registration is rejected.

**`src/core/` — portable RBAC engine.** This folder is deliberately project-agnostic (meant to be reusable across other Bun/Hono projects) and holds only the HTTP-layer enforcement mechanism, not this project's schema-coupled data access:
- `core/auth/auth.middleware.ts` — the JWT `authMiddleware` above.
- `core/rbac/rbac.middleware.ts` — `attachAuthz` (resolves a user's effective roles/permissions onto the context after `authMiddleware`, **and rejects suspended users with 403** — account `status` is embedded in the cached authz so a suspension takes effect on the caller's next request), `requirePermission(key)`, `requireRole(roleName)` (currently unused), and `enforceTenantScope(paramName?)` (compares a `:universityId`-style path param to the caller's own tenant; roles in `TENANT_SCOPE_BYPASS_ROLES` — `super_admin` and `platform_support` — bypass it; these bypass role names are this project's convention, called out as a portability caveat in a comment on the function).
- `core/rbac/guard.ts` — the single shared `guard(permissionKey, { tenantScoped? })` composer that every guarded route imports instead of redefining its own `[authMiddleware, attachAuthz, auditTrail(key), requirePermission(key)]` chain locally. Its return type is deliberately a fixed-length tuple (see the comment in the file) — adding/removing a middleware means updating `BaseGuardChain`/`TenantScopedGuardChain`.
- `core/rbac/audit-hook.ts` — the audit-trail hook inside `guard()`: `auditTrail(key)` wraps `next()` and, for mutating methods (non-GET), calls a project-registered sink with the final response status — so denied attempts (403) are recorded too, and a sink failure never fails the request. Core only defines the mechanism; the sink is injected at startup via `setGuardAuditSink` (same portability seam pattern as the rbac.cache read-back). Because it stays in `core/`, its own failure-to-write log line goes through its own `core/logger` instance rather than the app's `shared/logger` — see below.
- `core/logger/logger.ts` — a project-agnostic `createLogger(options)` pino factory (level/pretty/bindings only; never reads `env` or imports from `shared/`). It's the one exception to "core doesn't log": `audit-hook.ts` calls it directly to avoid a new core→shared coupling. Every other file builds its logger off the app's root instance instead (next paragraph).

The schema-coupled counterpart lives in `src/shared/rbac/` — `rbac.repository.ts` (computes effective roles/permissions **plus `status`** from `roles`/`rolePermissions`/`userPermissions`, tied to this project's exact Drizzle relations) and `rbac.cache.ts` (Redis read-through cache around it, 300s TTL, plus `invalidateUserPermissions`/`invalidateUsersPermissions`). Non-guarded routes that still need the suspension check (self-service `users`, `clubs`) apply `src/middlewares/active-user.middleware.ts` (`requireActiveUser`). Note the intended boundary is that `shared/rbac/*` imports only the `EffectivePermissions` contract type from `core/rbac/rbac.types.ts`; `core/rbac/*` does read back through `shared/rbac/rbac.cache.ts` (`getEffectivePermissions`) for the cached lookup, so the two are coupled at that one seam.

**Per-feature permission keys.** Each feature that guards routes owns a `<feature>.permissions.ts` file exporting an `as const` object of its permission-key strings (e.g. `AdminPermission.USER_VIEW`/`USER_MANAGE`, `ClubPermission.VIEW`/`APPROVE`/`MEMBER_MANAGE`, `UniversityPermission.FACULTY_CREATE`, `AuthPermission.ROLE_MANAGE`, `AnnouncementPermission.MODERATE`, `GalleryPermission.MODERATE`) plus, where seeded, a `*_PERMISSION_CATALOG` array of `{ key, description }`. Keys are granular `resource.action` and split read (`*.view`) from write so read-only roles (e.g. `auditor`) exist — do NOT gate a GET behind a write permission. These are a typo-safety convenience layer, not a closed set — `permissions` rows can still be created at runtime via the `role.manage`/`permission.manage` endpoints, so the DB stays the actual source of truth; `src/db/seed.ts` imports the same constants (into `ROLE_BUNDLES`) so seed data and route guards never drift apart. This convention covers the global RBAC layer (`admin`, `clubs`, `university`, `auth`, `announcements`, `gallery` features) — the per-club layer below is intentionally separate and doesn't use it.

**Background jobs**: `src/features/auth/auth.queue.ts` defines a BullMQ queue + worker that really sends the email-verification mail through `nodemailer` (`src/shared/mail/mailer.ts`, transport built from the `SMTP_*` env vars; the HTML/text template is `src/features/auth/auth.email.ts`). Locally `docker-compose` runs **Mailpit** as a fake SMTP server — it captures every message instead of delivering it and serves them at `http://localhost:8025`. Going to production changes only env vars, not code. Jobs retry 3× with exponential backoff; the Redis connection is derived from `env.REDIS_URL`. See `docs/MAIL_DOGRULAMA.md`.

**Notifications + WebSocket** (`src/features/notifications/`): notifications are persisted to a `notifications` table **and** pushed in real time to every device the user has open. Transport is Bun's native WebSocket via `hono/bun`'s `createBunWebSocket()` — called exactly once in `src/shared/ws/bun-ws.ts`, whose `websocket` handler must be passed to `Bun.serve` in `src/index.ts` (under the `import.meta.main` entrypoint guard) or upgrades silently break. Handshakes can't carry an `Authorization` header, so auth uses a single-use 60s **ticket** (`POST /api/notifications/ws-ticket` → Redis `SETEX`; the upgrade consumes it with `GETDEL`) rather than a JWT in the query string. Cross-instance delivery goes through Redis Pub/Sub on the `notifications` channel — the subscriber needs its **own** connection (`shared/redis/redis.subscriber.ts`), because ioredis forbids normal commands on a connection in subscriber mode and would otherwise break the RBAC cache and rate-limit counters. Feature code should call `notificationsService.notifySafe(...)`, which logs and swallows failures: a club approval must not fail because a notification couldn't be delivered. The `notifications.type` column is a plain `varchar`, with the `NotificationType` `as const` catalog in `notifications.types.ts` acting as a typo-safety layer (same convention as `*.permissions.ts`) so new types don't need a migration.

**Rate limiting** (`src/middlewares/rate-limit.middleware.ts`): Redis fixed-window counters, fail-open if Redis is down. Keyed by the **identity of the protected resource, not the IP** — students share one campus NAT IP, so an IP-keyed limit would lock out the whole university. `login` and `resend-verification` are keyed by email; only `register` (where no identity exists yet) uses a generous IP ceiling.

**Account status gating**: `requireActiveUser` (`middlewares/active-user.middleware.ts`) rejects `suspended`; `requireVerifiedUserForWrites` (`middlewares/verified-user.middleware.ts`) rejects `pending` on non-GET methods only, mounted once at the `clubs` router root so new write routes are covered automatically. Unverified users can log in and read on purpose (the frontend's "verify your email" banner and resend flow need a session). Both read `status` from the RBAC cache, so `authService.verifyEmail` and `adminService.updateUserStatus` must call `invalidateUserPermissions` — otherwise the change is invisible for the 300s TTL. Denials carry a machine-readable `code` (`EMAIL_NOT_VERIFIED`, `RATE_LIMITED`) so the frontend never string-matches messages.

**Error handling**: services signal business-rule failures by throwing a **plain** `new Error("Turkish user-facing message")` — this is a contract, not a shortcut. Both `errorHandler` (`src/middlewares/error.middleware.ts`) and `respondWithBusinessError` (`src/shared/utils/error.util.ts`) distinguish these from infrastructure errors via `err.constructor === Error` (pg throws `DatabaseError`, drizzle `DrizzleQueryError`, the runtime `TypeError` — always subclasses). Business errors return 400/404 with their message; everything else returns a **generic 500** (never leak SQL/table names to clients) and is logged server-side with the request's `requestId` (`hono/request-id`, echoed in every error response for correlation). Route `catch` blocks must call `respondWithBusinessError(c, error, statusFromError?)` — it rethrows non-business errors into `app.onError` — instead of blindly returning `error.message`.

**Structured logging** (`src/shared/logger/logger.ts`): the app's single root pino logger, built via `core/logger/logger.ts`'s portable factory with level/pretty-print picked from `NODE_ENV`/`LOG_LEVEL` (prod: `info` + raw JSON for log aggregators; dev: `debug` + colorized `pino-pretty`). Every other file gets its own scoped logger with `logger.child({ module: "..." })` — never call `console.*` in request-path or background-job code. `src/middlewares/request-logger.middleware.ts` replaces `hono/logger()`: one JSON line per request (`requestId`, method, path, status, `durationMs`), leveled by status (5xx → error, 4xx → warn). `error.middleware.ts` logs unexpected errors the same way, keyed by the same `requestId` so a user-reported failure can be matched to its server-side log line and stack trace. `db/seed.ts` and `db/sync-permissions.ts` are deliberately exempt — they're one-off CLI scripts meant to be read by a developer in a terminal, not aggregated. The request logger is the portable `core/http/request-logger.ts` factory (status→level, message, and extra-field seams injected); the project setup (`request-logger.middleware.ts`) injects Turkish messages and enriches each line with `userId`/`universityId` (read after `next()`, once auth has run). The root logger **redacts** sensitive fields (`password`, `token`, `authorization` → `[Redacted]`) and, when `LOG_FILE` is set, mirrors raw JSON to a file in addition to stdout. An optional self-hosted **Vector → Loki → Grafana** observability stack (in `docker-compose.yml`, `docker compose up -d`, Grafana at `:3001`) ships and searches those logs — see [docs/LOGLAMA.md](docs/LOGLAMA.md).

**Audit trail** (`src/features/audit/`): every mutating request through `guard()` is automatically recorded to the append-only `audit_logs` table (actor, action = permission key, method/path/status, derived target, redacted body, ip) by the sink in `audit.sink.ts` (registered in `index.ts` via `registerAuditSink()`). There are deliberately **no write/delete endpoints** — only `GET /api/audit/universities/:universityId` (cursor-paginated like notifications), gated behind `audit.view` (held by `auditor`, `university_admin`, `platform_support`, `super_admin`). Denied attempts land in the log with status 403; suspended users are cut before the hook, so they don't. The per-club management layer (`club.middleware`) does not go through `guard()` and is therefore *not* auto-audited. `db:sync-permissions` knows how to backfill `audit.view` into an existing DB.
